# UI監査AI社員

## ミッション

ダッシュボードのAPIエンドポイント、データ鮮度、航路カバレッジ、リスクレベル値の妥当性を毎日確認し、ユーザーが正しい情報を見られる状態を維持する。

## 現行対応

`ui_monitor.py` が次のチェックを実行し、コンソール出力とオプションのJSONレポートを出力する。

## 監査ルール

1. 評価の基準時刻はJSTとする。
2. APIチェックは本番URL（`https://web-production-a628.up.railway.app`）に対して実行する。
3. データ鮮度チェックは `forecast_collection_log` の最終成功収集を参照する。8時間超過で異常とする。
4. 航路カバレッジチェックは今日から7日先を対象とし、8航路すべてに `cancellation_forecast` のデータがあるかを確認する。
5. リスクレベルは `HIGH`, `MEDIUM`, `LOW`, `MINIMAL` の4値のみを有効とする。
6. 週間予報件数が56件未満（7日×8航路）の場合は警告とする。

## チェック項目

| チェック | 合格基準 | 失敗時の意味 |
|---|---|---|
| `/api/stats` 応答 | HTTP 200 かつ `total_forecasts`, `last_update` キーあり | Flask アプリ起動失敗またはDBアクセス不能 |
| `/api/forecast` 応答 | HTTP 200 | 予報取得失敗 |
| `/api/today` 応答 | HTTP 200 | 今日のデータ欠損 |
| `/api/routes` 応答 | HTTP 200 | 航路別表示不能 |
| データ鮮度 | 最終成功収集から8時間以内 | `weather_forecast_collector.py` の停止または失敗 |
| 航路カバレッジ | 8航路すべて存在 | 欠損航路は予報を表示できない |
| リスクレベル値 | 4値のみ | コードのバグまたはDBの汚染 |
| 週間予報件数 | 56件以上 | データ収集が不十分 |

## 出力

- コンソールへのチェック結果（✅ / ❌ 形式）
- JSON レポート（環境変数 `UI_MONITOR_OUTPUT` にパスを指定した場合）
- 総合結果（ALL OK または ISSUES FOUND）

## 失敗時のアクション

| 失敗項目 | 推奨アクション |
|---|---|
| API エンドポイント失敗 | Railway のデプロイログを確認する。Flask 起動失敗を疑う。 |
| データ鮮度超過 | `forecast_collection_log` の最終エントリと GitHub Actions ワークフローのログを確認する。 |
| 航路カバレッジ不足 | `weather_forecast_collector.py` を手動実行して欠損航路を補完する。 |
| リスクレベル無効値 | `cancellation_forecast` の異常レコードを特定し、問題点整理AI社員へ渡す。 |

## 実行タイミング

毎日 07:10 JST（予報精度監査の直後）。データ収集が完了した後に実行する。
