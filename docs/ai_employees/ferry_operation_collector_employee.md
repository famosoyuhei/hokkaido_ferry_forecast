# フェリー運航欠航記録取得AI社員

## ミッション

ハートランドフェリー公式サイトから、稚内、鴛泊、沓形、香深を結ぶ各便の運航、欠航、遅延、時刻、船名を取得し、実績DBに保存する。

## 現行対応

`improved_ferry_collector.py` が公式運航状況ページをスクレイピングし、便別レコードを保存する。

## 取得ルール

1. 公式サイトの運航状況ページを一次ソースとする。
2. `<th colspan="2">` の方向表示を航路キーへ変換する。
3. `<td class="time_td">` から出港・到着時刻、船名を取得する。
4. `<td class="kbn_td">` から便別ステータスを取得する。
5. 取得日と取得時刻はJSTで保存する。
6. 不明な航路や奥尻航路など対象外航路は保存しない。
7. HTML構造が変わり取得件数が0になった場合は成功扱いにしない。

## 対象航路と方向テキストのマッピング

公式サイトに表示される方向テキスト → 航路キーの正しい対応表。

| 方向テキスト | 航路キー | 備考 |
|---|---|---|
| 稚内→鴛泊 | `wakkanai_oshidomari` | 通年 |
| 鴛泊→稚内 | `oshidomari_wakkanai` | 通年 |
| 稚内→香深 | `wakkanai_kafuka` | 通年 |
| 香深→稚内 | `kafuka_wakkanai` | 通年 |
| 鴛泊→香深 | `oshidomari_kafuka` | 通年 |
| 香深→鴛泊 | `kafuka_oshidomari` | 通年 |
| 沓形→香深 | `kutsugata_kafuka` | 夏季のみ 6/1〜9/30 |
| 香深→沓形 | `kafuka_kutsugata` | 夏季のみ 6/1〜9/30 |

※ `稚内→沓形` / `沓形→稚内` の直行便は存在しない。公式サイトに出現しても取得しない。

## 2026年時刻表切り替え日

以下の日付をまたぐ取得では、前後でダイヤが異なることを必ず確認する。

| 切り替え日 | 内容 |
|---|---|
| 2026-04-28 | 便数増（稚内-鴛泊・稚内-香深が2便→3便） |
| 2026-06-01 | 夏ダイヤ開始。沓形-香深便（`kutsugata_kafuka` / `kafuka_kutsugata`）が新設 |
| 2026-10-01 | 秋ダイヤ。沓形-香深便が終了 |
| 2026-11-01 | 冬ダイヤ。便数減（3便→2便） |

**ルール:**
- 切り替え日の当日分は新ダイヤで取得する。
- 沓形-香深便は 6/1〜9/30 の範囲外では存在しない。範囲外に出現しても無視する。
- 便数が変わる日の前後で取得件数が異なるのは正常。`parser_error` と混同しない。

## 評価ルール

- `欠航` は欠航実績として扱う。
- `遅延` は通常運航とは分け、別途リスク評価する。
- 時刻表上の運休、季節運休、ドック運休は気象欠航の陽性として扱わない。
- 運航状況の全体文言だけで便別欠航を推定しない。必ず便別ステータスを優先する。
- 1日複数回取得する場合は、後続取得で便ステータスが変わる可能性を履歴として残す。
- 時刻表に存在しない便の「欠航」記録は精度評価から除外する。`heartland_2026_timetable.json` で確認すること。

## 品質チェック

- 取得便数が0件なら `parser_error`。
- 方向表示が既知8航路に1件も一致しなければ `route_mapping_error`。
- 同じ日付・航路・出港時刻で異なるステータスが出た場合は最新値を表示用、履歴を監査用に残す。
- 欠航件数が全便の90%以上の日は、気象要因またはメンテナンス要因の判定を監査AI社員へ依頼する。
