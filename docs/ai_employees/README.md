# AI社員設計: 稚内・利尻・礼文フェリー欠航予報

このディレクトリは、将来オートメーション化する前提で、フェリー欠航予報システムのAI社員、取得ルール、評価ルール、監査ルール、修正依頼プロンプト生成ルールを定義する。

## AI社員一覧

| AI社員 | 目的 | 現行対応ファイル | 将来の自動化単位 |
|---|---|---|---|
| 海上気象予報取得AI社員 | 各航路・各航行時間の予報気象を取得し、欠航リスク入力を作る | `weather_forecast_collector.py` | 1日4回以上 |
| 海上気象実測取得AI社員 | 各港・航行時間帯の実測/再解析気象を取得する | `actual_weather_collector.py` | 1日1回、前日分 |
| 予報精度監査AI社員 | 予報、実測、運航実績を突合し、精度と失敗パターンを監査する | `unified_accuracy_tracker.py` | 1日1回 |
| フェリー運航記録取得AI社員 | ハートランドフェリーの便別運航/欠航記録を取得する | `improved_ferry_collector.py` | 1日1回以上 |
| 問題点整理・修正依頼プロンプトAI社員 | 監査結果から修正候補を整理し、実装依頼プロンプトを作る | `issue_prompt_composer.py` | 監査後、週1回または異常時 |
| 欠航リサーチAIスキル | 気象・航路・運航判断に関する調査と知識更新を支援する | `skills/ferry-cancellation-research/` | 必要時、改善検討時 |

## 共通原則

- 時刻は必ずJSTで扱う。
- 対象港は稚内、鴛泊、沓形、香深の4港。
- 対象航路は稚内-鴛泊、鴛泊-稚内、稚内-香深、香深-稚内、稚内-沓形、沓形-稚内、鴛泊-香深、香深-鴛泊の8方向。
- 予報と実測は「日別」だけではなく「便別・航行時間別」に保存または再構成できる粒度を保つ。
- 欠航判定は単純な風速だけでなく、波高、視程、季節、航路差、港差を必ず分けて評価する。
- データ欠損、取得失敗、推定値の混入は通常値として扱わず、監査対象にする。

## 成果物の標準形

各AI社員は次を出力する。

- 実行日時、対象日、対象航路/港、データソース
- 取得件数、欠損件数、異常件数
- DBへ保存したテーブル名と主キー
- 失敗時の原因分類
- 次のAI社員が読むべき要約

## 自動化前の推奨順序

1. 現行コードの時刻、港名、航路名、DBスキーマをこの設計に合わせる。
2. 便別の航行時間帯に予報/実測を割り当てる中間テーブルを作る。
3. 監査AI社員がFalse Negative/False Positiveを便別に説明できるようにする。
4. 問題点整理AI社員が修正依頼プロンプトを生成する。
5. Railway CronまたはCodex Automationへ段階的に移す。
